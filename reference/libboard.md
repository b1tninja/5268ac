# `libboard.so` — board identity, secrets, and access-code inputs

Firmware slice: **`att-5268-11.5.1.532678_prod_lightspeed-install`** (MIPS big-endian). Analysis uses Ghidra project **`5268ac`**, program **`/usr/lib/libboard.so.0.0.0`** (~144 analyzed functions, ELF base **`0x00010000`**).

This note documents how **`libboard`** exposes **serial number**, **access/auth/subscriber codes**, and related keys — material that **feeds higher layers** (including **`httpd`**) for **login / subscriber access-code flows**. It complements **[`cm_cmdb.md`](cm_cmdb.md)** (CMDB runtime): **`libboard`** is **not** the CMDB transaction API; it is the **manufacturing / board-parameter** layer that reads **stable device identity** and **key blobs**, typically backed by **files or param stores** on a **writable filesystem** (often UBIFS on MTD — see **NAND correlation** below).

## Exported API groups (symbol-level)

### Build / capability (context only)

- **Build strings:** `board_build_machine`, `board_build_user`, `board_build_time`, `board_build_ostype`, `board_build_osrevision`, `board_build_version`, `board_build_version_hash`, `board_build_display`, `board_get_deferred_version`, `board_build_digits`, `_board_build_version`
- **Feature flags:** `feature_exist`, `get_usb_ol`, `has_hpna`, `has_battery`, `has_5g`, `has_quanwifi`, `has_bcmwifi`, `has_voice`, `has_bonded`, `has_atm`

### Identity fields read as files (`board_info_*`)

Many **`board_info_*`** routines follow the same idiom (verified in Ghidra for **`board_info_serialnumber`** and **`board_info_accesscode`**):

1. **Two candidate path pointers** (plus terminator) in a small array.
2. **`open()`** each path until one succeeds (fallback layout).
3. Call **`_board_read_parameter(fd, user_buffer)`** to populate the caller’s buffer.
4. **`close()`** the descriptor.

**Implication:** **Serial number** and **printed access-code source material** are treated like **small text files** on whatever filesystem is mounted at runtime — **not** raw NAND IOCTLs inside **`libboard`**. Any **NAND** involvement is **below** this layer (block device → filesystem → path).

Addresses (for re-check in Ghidra): **`board_info_serialnumber`** @ **`0x00011f84`**, **`board_info_accesscode`** @ **`0x00012420`**.

### Key / code material (`board_key_*`)

| Symbol | Role (from decompilation summary) |
|--------|-------------------------------------|
| **`board_key_accesscode`** | Zeros **`0x84`**-byte output; reads **`0x80`** bytes into trailing buffer via internal reader; stores **length** in first word. Same structural pattern as **`board_key_authcode`**. |
| **`board_key_authcode`** | Same **`0x84`** layout: length word + **`0x80`** payload from internal reader. |
| **`board_key_systemcode`** | Builds **two 16-digit decimal ASCII strings** (digits **`0`–`9`**) from repeated **mix/compress** steps over buffers — **`% 10` then `+ '0'`** — after concatenating fixed template bytes (**`snprintf`**/ **`sprintf`**-style prefix **`0x11`** bytes from rodata relative to GP). Uses MD5 imports (`MD5_*`) elsewhere in the shared object; treat as **cryptographic derivation of printable codes** from **board secrets + fixed strings**. Address **`0x00013b54`**. |
| **`board_key_secret`** | Parses an **`0x80`**-char **hex-or-digit string** into binary (**ctype** tables), writes **`0x80`** bytes into output (**pairwise hex decode**). Address **`0x000139c0`**. |
| **`board_key_wireless`** | Same family as **`board_key_accesscode`** (length + **`0x80`** payload pattern). |

These routines align with **operator “subscriber code” / “access code”** generation: **deterministic numeric codes** derived from **device-bound secrets**, not the CMDB user table.

### Parameter store (`board_param_*`)

- **`board_param_open`**, **`board_param_get`**, **`board_param_set`**, **`board_param_exist`**, **`board_param_size`**, **`board_param_clear`**, **`board_param_remove`**, **`board_param_show`**, **`board_param_flush`**, **`board_param_close`**

**`board_param_get`** @ **`0x00014848`** delegates to an internal **`param_get`** then checks buffer size (**`0x22`** if too small — **`EINVAL`**-style). This is the **structured board-parameter database** (opened by **`board_param_open`**, which **`read()`**s a blob and validates length — file-backed store).

### Integrity

- **`uboot_crc32`** — CRC32 helper (image / blob integrity; often paired with **U-Boot environment** or **manufacturing blobs**).

## How this relates to **NAND**

What we can state from **`libboard`** alone:

1. **No direct `ioctl(MEMREAD)` / MTD raw access** appears in the **import table** for this SO — reads use **`open64`**, **`read`**, **`stat64`**, **`statvfs64`**, **`sysinfo`** — i.e. **POSIX file semantics**.
2. On this class of gateway, **writable state** (including paths like **`/rwdata/...`** referenced elsewhere in the repo — see **`cm_cmdb.md`** / **`fwupgrade.txt`**) normally lives on **UBIFS** or similar **over NAND MTD** partitions.
3. **`pace_tl_map.json`** and **`opentl_kernel_ghidra.md`** describe **OpenTL / BBM** and **carving** raw dumps — use those when correlating **offline NAND images** to **runtime paths**, not **`libboard`** strings (paths are not reliably visible as plain ASCII in the **`libboard`** **`strings`** sweep; they are typically **GP-relative pointers** in MIPS).

**Working model:** **NAND → MTD → UBIFS (or JFFS2) → VFS path → `libboard` `open`/`read`.** To recover **exact pathnames** for serial/access templates, use **runtime evidence** (**`strace`** on **`httpd`** calling **`board_info_serialnumber`**) or **dump `.rodata`** in Ghidra at the pointers referenced from **`board_info_*`** (candidate path table).

## Hypothesis: access-code generation chain

Not proven end-to-end in one binary, but consistent with imports and decompilation:

1. **`libboard`** reads **serial** / **secret key material** / **subscriber buffers** from **files** and **`board_param_*`**.
2. **`board_key_systemcode`** (and related **`board_key_*`**) **derive decimal codes** from those secrets.
3. **`httpd`** composes **HTML/API responses** and compares user-entered codes against derived or stored values; **CMDB / `tw_ulib_pwd_*`** handles **account passwords** separately (**[`cm_cmdb.md`](cm_cmdb.md)**, **`output/tw_ulib_pwd_re.md`**).

## Ghidra follow-ups (concrete)

1. **`_board_read_parameter`** — rename/locate internal symbol; confirm **`read()`** loop and newline stripping.
2. **Rodata dump** at **GP-relative** bases used by **`board_info_serialnumber`** / **`board_info_accesscode`** — recover **both fallback path strings**.
3. **`httpd`** — xref **`board_key_systemcode`** / **`board_key_accesscode`** into CGI / HURL handlers (see **`httpd_endpoints.md`**) to map **UI fields** ↔ **`libboard`** calls.
4. **`modprov`** — xref **`board_key_*`** for **provisioning** paths that might log or reset codes.
